Базовые принципы проектирования

Андрей Воронин

Выполняя базовые принципы проектирования, дизайнеры взаимодействия и разработчики могут создавать системы, отвечающие ожиданиям человека.

Такие системы люди будут считать естественными.

3.

Скромность

Скромность

Скромность

Выбором места хранения файла
должен заниматься компьютер

Плохо

Загрузка в Firefox

Хорошо

Загрузка в Chrome

Вывод

  1. Не спрашивайте и не ждите действия человека,
    когда он попросил что-то сделать.
  2. Делайте предварительную загрузку.
  3. Для временного хранения файлов используйте
    специальную папку.

Скромность

Созданием резервной копией
должен занимать компьютер даже
без первичной настройки человеком

Плохо

File History из Windows

Хорошо

Version history из Dropbox

Вывод

  1. Делайте бекапы без предварительной настройки.
  2. Иметь версию файла, актуальную неделю назад, такое же базовое требование приложения, как устойчивость к сбоям.
  3. Сохраняйте инкрементальные изменения.
  4. Человек вспомнит о бекапе только тогда, когда будет уже поздно.
  5. Плохой бекап лучше, чем никакой.

Скромность

Защитой и обслуживанием компьютера должен заниматься он сам

Плохо

Kaspersky Internet Security

Хорошо

Windows Defender Antivirus

Вывод

  1. Безопасностью компьютера должен заниматься компьютер.
  2. Если найден вредоносный файл, можно показать ненавязчивое уведомление (хорошее решение).
  3. Или обновить базу, проверить автоматически диск, и сказать что “Вы в безопасности” (лучше).
  4. Или тихо нейтрализовать вирусы (великолепное решение).

Скромность

Организацией закладок
должен заниматься компьютер

Плохо

Закладки в Chrome

Хорошо

нет примера

Вывод

  1. Сохранение закладок должно быть в 1-клик без запросов.
  2. Категоризация и установка тегов должна быть автоматическая
    по содержимому страницы.
  3. Нужно сохранять вспомогательную информацию, которая поможет
    при последующем поиске.

Итого по скромности

  1. Вопросы и уведомления должны касаться только целей человека.
  2. Не путайте цели человека и цели компьютера.
  3. На вопросы, не касающиеся целей человека, будут даны
    вредные и опасные ответы.
  4. Оценивая функциональность приложения считайте, что вопросы, необходимые компьютеру, будут проигнорированы.

Надежность

Надежность

Надежность

При закрытии нужно сохранять документ всегда, независимо
от действий человека

Плохо

Cохранение в TextEdit

Хорошо

Сохранение в Google Docs

Вывод

  1. Сохраняйте данные, независимо от действий человека.
  2. Даже, если он явно сказал, что документ ему не нужен.

Надежность

Сохранять нужно все данные, полученные от человека

Плохо

Прокрутка в Word Online

Хорошо

Прокрутка в Word 2013

Вывод

  1. Сохраняйте не только основной документ,
    но и всю вспомогательную информацию,
    которая может помочь в будущем.

Надежность

Хранить нужно не только последнюю версию файла

Плохо

Автоматическое сохранение в Notes.app

Хорошо

Version history в Google Docs

Вывод

  1. Хранить нужно не только последнюю версию файла,
    но и предыдущие.
  2. Предыдущие версии файла так же ценны.

Итого по надежности

  1. Данные человека бесценны.
  2. Данные человека нужно сохранять все и всегда.
  3. Данные человека – это не только написанное сообщение.
  4. Хранить нужно даже то, что человек явно удалил.
  5. Хранить нужно не только последнюю версию файла.
  6. Для хранения используйте контроль версий.

1.

Отзывчивость

Отзывчивость

Отзывчивость

Закрытие приложений должно
быть мгновенным, независимо
от обстоятельств

Плохо

Закрытие приложений во “взрослых”
операционных системах

Хорошо

Закрытие приложение в мобильных операционных системах

Вывод

  1. Приложения должны быть рассчитаны, что их в любой момент могут закрыть.
  2. При закрытии приложение не должно делать запросов у человека.
  3. “Закрыть” ≠ “Завершить работу приложения”.

Отзывчивость

Реакция сетевых приложений
должна быть мгновенной и позитивной

Плохо

Переключение на следующее фото в Twitter

Хорошо

Like в Facebook

Вывод

  1. Cетевые приложения должны понимать, что интернет
    может быть плохой.
  2. И это не крайний случай, это обычная жизнь приложения.

Отзывчивость

Появление контента
должно быть мгновенным

Плохо

Открытие приложения Twitter

Хорошо

Загрузка поста в Facebook

Вывод

  1. При загрузке сначала показывайте placeholder контента.
  2. Изображения — JPEG с прогрессивной разверткой.
  3. Текст — placeholder, показывающий структуру сообщения.
  4. Приложения — не используйте экран загрузки для брендинга.

Итого по отзывчивости

  1. Отделяйте представление приложения от его логики.
  2. Внезапное закрытие, отсутствие сети, нехватка ресурсов –
    это нормальные условия работы приложения.
  3. Если человек попросил что-то сделать, не нужно спрашивать.
  4. Реакция в сетевых приложениях должна быть позитивной.
  5. Показывайте placeholder, пока контент не был получен.
  6. Используйте предварительную загрузку содержимого.

Общий итог

Общий итог

При создании приложений пользуйтесь базовыми принципами проектирования для создания естественного взаимодействия
и удовлетворенности от использования.

  • Отзывчивость создаст удобное использование.
  • Благодаря надежности человек получит безопасную систему.
  • В следствии скромности компьютера человек будет полностью погружен в решении своих задач.
